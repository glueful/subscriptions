# Subscriptions v1.1 -- Managed Plan Catalog -- Design Note

**Status:** Draft for redline.
**Date:** 2026-06-10
**Repo:** `glueful/subscriptions` (standalone extension).
**Companion:** Builds on the locked and implemented Subscriptions v1 design (`docs/superpowers/specs/2026-06-09-subscriptions-v1-design.md`). This is additive: v1's checker, middleware, lifecycle projection, overrides, and Payvia integration stay intact.

---

## Why this spec exists

Subscriptions v1 deliberately made the entitlement catalog config-driven: plans live in `config/subscriptions.php`, the catalog is versioned with code, and deployments atomically change gates and grants together. That was a safe v1 sequencing choice, not a claim that config is the permanent product answer.

`glueful/subscriptions` now has a concrete consumer that needs non-developer plan management: Lemma. Not all Lemma operators are developers, and a SaaS admin UI needs to create, update, archive, and inspect commercial plans without editing PHP config or waiting for a deploy. That consumer is enough to promote the catalog to a DB-managed product surface.

The v1 architecture already prepaid most of this change: everything downstream consumes a `PlanCatalog`-style resolver. v1.1 makes that resolver two-source:

```text
config plans = seed data / fallback / disaster safety
DB plans     = app-managed commercial catalog
overrides    = per-tenant exceptions
```

## Goals

1. Add a managed `subscription_plans` table for plan definitions and entitlement grants.
2. Keep config plans as seed/fallback, not as the only catalog source.
3. Extend PlanCatalog resolution so DB plans win over config for active/archived rows.
4. Add a permission-gated HTTP management API for plan CRUD-style operations.
5. Add CLI commands for create/update/archive/import-config.
6. Audit plan changes with old/new entitlement diffs.
7. Preserve all existing v1 runtime consumers: checker, middleware, rate-tier bridge, lifecycle, overrides, and Payvia event projection.

## Non-goals

- Usage metering / quota consumption. Numeric limits remain static grants; counters/windows are still separate.
- Versioned plan revisions / immutable historical catalog snapshots.
- Checkout / upgrade UX.
- A generic CMS for arbitrary commercial objects.
- Changing `subscriptions.plan_key`; it continues to point at a stable `plan_key`.
- Replacing per-tenant overrides.
- Making Payvia required.

## Core model

### New table: `subscription_plans`

| Column | Type | Notes |
|---|---|---|
| `id` | bigint pk autoincrement | |
| `uuid` | `string(12)` unique | Internal NanoID, generated by this extension. |
| `plan_key` | `string(64)` unique | Stable key such as `free`, `pro`, `enterprise`; joins to `subscriptions.plan_key`. |
| `display_name` | `string(120)` | Admin/UI label. |
| `description` | `string(255)` nullable | Admin/UI help text. |
| `entitlements` | json | Same value space as config: `bool`, `int >= 0`, or explicit `null`. |
| `payvia_priced_plan_uuid` | `string(12)` nullable | FK-less link to Payvia priced plan. |
| `status` | `string(20)` | `draft`, `active`, `archived`. |
| `sort_order` | int default `0` | Admin/UI ordering. |
| `created_at` / `updated_at` | timestamps | |

No FK to Payvia. `payvia_priced_plan_uuid` is an opaque, nullable bridge exactly like the existing Subscriptions/Payvia soft-link posture.

Migration mechanics: add `004_CreateSubscriptionPlansTable.php` under the Subscriptions migration set, register it with `MigrationPriority::DEPENDENT`, source `glueful/subscriptions`, and follow the existing house convention: extension-owned `uuid` columns are 12-character NanoIDs while external/soft-link keys remain opaque strings.

The migration must use the real framework migration contracts:

```php
use Glueful\Database\Migrations\MigrationInterface;
use Glueful\Database\Schema\Interfaces\SchemaBuilderInterface;
```

It must also follow the existing Subscriptions migration guard pattern: return early when the table already exists, default `created_at` to `CURRENT_TIMESTAMP`, and keep `updated_at` nullable.

### Status semantics

- `draft`: editable, not assignable, not used for entitlement resolution.
- `active`: assignable and resolvable.
- `archived`: not assignable to new tenants, but still resolvable for existing tenants.

Archived is not deleted. A tenant already on an archived plan must keep resolving that plan, otherwise archiving a plan would silently strand paying tenants or downgrade them unexpectedly.

Draft is a pre-publish state only. Legal status transitions are:

| From | To | Allowed? | Reason |
|---|---|---:|---|
| `draft` | `active` | yes | Publish a new plan. |
| `draft` | `archived` | yes | Abandon an unpublished plan. |
| `active` | `archived` | yes | Retire without stranding existing tenants. |
| `archived` | `active` | yes | Un-archive and make assignable again. |
| `active` | `draft` | no | Would stop resolving the DB row and can strand tenants. |
| `archived` | `draft` | no | Would stop resolving the DB row and can strand tenants. |

The API/CLI validation layer must reject `active -> draft` and `archived -> draft`. This rule is unconditional in v1.1, not only when subscriptions currently reference the plan, because a DB-only plan could be assigned later through stale state or external tooling and the safer invariant is simple: once published or archived, never return to draft.

Deletion is intentionally absent in v1.1. Plan lifecycle is create -> draft/active -> archived, with optional archived -> active restoration.

## Entitlement value semantics

Managed DB plans use the same value space as config plans and the v1 checker:

| Value | `allows()` | `limit()` | Meaning |
|---|---:|---:|---|
| key absent | `false` | `0` | Undefined key denies. |
| `false` | `false` | `0` | Explicitly disabled. |
| `0` | `false` | `0` | Numeric disabled/zero. |
| positive int | `true` | int | Bounded grant. |
| `true` | `true` | `null` | Unlimited boolean grant. |
| explicit `null` | `true` | `null` | Unlimited numeric grant. |

This is the most important validation rule in the write surface. The DB API must reject strings, floats, arrays, negative ints, and objects as entitlement values because those would corrupt deny-on-absent semantics.

Undefined entitlement key still denies. Explicit `null` is the only unlimited numeric grant.

An empty entitlement map (`{}`) is valid. For an active plan it means "deny every entitlement key" because absent keys deny. This is useful for suspended, internal, or parking tiers; it should be explicit in admin UI copy so operators do not mistake it for "inherit from config."

## PlanCatalog resolution

`PlanCatalog` becomes a two-source catalog. Everything downstream still calls the catalog; no checker/middleware/lifecycle code reads `subscription_plans` directly.

### `entitlementsFor(string $planKey): array`

Resolution order:

1. Look for a DB row by `plan_key`.
2. If DB row exists and `status in ('active', 'archived')`, return its `entitlements`.
3. Otherwise, fall back to config `plans[$planKey]['entitlements']`.
4. If neither source has the plan, return an empty map. The checker's existing key-level absent-deny behavior handles the result.

Draft rows do not resolve. This prevents an operator from accidentally granting an unfinished plan.

### Assignment rule

Assignment/start-plan flows may assign only `active` DB plans or config fallback plans. They must reject `draft` and `archived` DB plans for new assignment.

Name the rule in one place:

```php
PlanCatalog::isAssignable(string $planKey): bool
```

`isAssignable()` returns true when there is an active DB row for the key, or when there is no DB row and a config plan exists for the key. `SubscriptionService::start()`, `SubscriptionService::changePlan()`, `subscriptions:set-plan`, and any HTTP/admin assignment flow must call this method instead of re-implementing assignment rules.

Existing tenants on archived plans are not reassigned by archive. They continue to resolve the archived DB row.

### `version(): string`

The catalog fingerprint becomes:

```text
sha1(json_encode(config('subscriptions.plans'))) . ':' . max(subscription_plans.updated_at)
```

If the DB table is empty, the DB part is `none`.

The existing resolution cache already includes the catalog version. This change makes plan edits propagate on the next resolve without opcache concerns or file writes. No new cache invalidation bus is required for v1.1.

If the extension is enabled before migration 004 has run, catalog resolution must not throw. `PlanCatalog` should catch the missing-table/database failure around DB lookups and `max(updated_at)`, then behave as config-only. This preserves the v1 guarantee that config-only deployments keep working until migrations are applied.

Two operational details:

- `max(updated_at)` has one-second resolution on common SQL backends. Two edits in the same second may share a fingerprint, so stale entitlement resolution can last until the existing cache TTL expires (default 300 seconds). This is acceptable for v1.1 because plan writes are low-frequency admin operations; immutable revisions or monotonic catalog versions are deferred.
- Every cache-key build performs a cheap `SELECT MAX(updated_at)` against a table expected to contain dozens or hundreds of rows, not millions. Add an index on `updated_at` anyway; it is inexpensive and makes the assumption explicit.

### Config stays operational

These stay config-only in v1.1:

- `default_plan`
- `grace_days`
- cache settings
- rate-tier ordering/settings

They are operational knobs, not catalog content.

## HTTP write surface

All endpoints are admin/commercial control-plane operations and fail closed. They require this permission:

```text
subscriptions.plans.manage
```

Do not rely on framework `gate_permissions` for v1.1 enforcement. As of this design, `GateAttributeMiddleware` depends on request `handler_meta`, and the framework router does not populate that attribute for matched handlers. That is a separate framework security fix candidate for 1.54.1 and affects existing users routes too. Subscriptions v1.1 must enforce directly through a subscriptions-owned guard that calls `PermissionManager::can()`.

Enforcement shape:

- Routes use `auth` plus a new `subscriptions_plans_manage` middleware alias.
- `subscriptions_plans_manage` reads the authenticated `auth.user`, resolves `permission.manager` / `PermissionManager`, and calls `can($userUuid, 'subscriptions.plans.manage', 'subscriptions.plans', $context)`.
- Missing user, missing manager, missing provider, or denied permission returns 403.
- The controller may document permission intent, but the middleware is the enforcement point.

Routes:

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/subscriptions/plans` | List plans. |
| `GET` | `/subscriptions/plans/{key}` | Show a plan by `plan_key`. |
| `POST` | `/subscriptions/plans` | Create a plan. |
| `PATCH` | `/subscriptions/plans/{key}` | Update display fields, entitlements, status, Payvia link, sort order. |
| `POST` | `/subscriptions/plans/{key}/archive` | Archive a plan. |
| `POST` | `/subscriptions/plans/import-config` | Seed DB rows from config plans. Optional HTTP mirror of the CLI import. |

Route ordering must register `/subscriptions/plans/import-config` before `/subscriptions/plans/{key}`. Also reserve `import-config` as an invalid `plan_key` so the route and a legal key cannot collide conceptually.

Because `plan_key` allows dots, every `{key}` route must add an explicit route constraint:

```php
->where('key', '[a-z0-9._-]+')
```

Create payload:

```json
{
  "plan_key": "pro",
  "display_name": "Pro",
  "description": "For growing teams",
  "entitlements": {
    "reports.export": true,
    "projects.limit": 50,
    "api.monthly": 100000
  },
  "payvia_priced_plan_uuid": null,
  "status": "active",
  "sort_order": 20
}
```

Patch payload uses the same fields, all optional except no `plan_key` rename in v1.1. Plan keys are stable because `subscriptions.plan_key` points at them.

### Validation

`PlanRequest` / DTO rules:

- `plan_key`: required on create; string; 1-64; lowercase letters, numbers, dot, underscore, hyphen; unique on create; immutable on patch.
- `display_name`: required on create; string; 1-120.
- `description`: nullable string; max 255.
- `entitlements`: required on create; object/map; each key string max 128; each value is `bool`, `int >= 0`, or `null`.
- `payvia_priced_plan_uuid`: nullable string length 12; soft-check if Payvia is installed, but no hard dependency.
- `status`: one of `draft`, `active`, `archived`.
- `sort_order`: integer.

Additional validation:

- `plan_key` must not equal reserved words used by collection actions, currently `import-config`.
- `entitlements` may be an empty object/map; when empty, the plan denies every undefined entitlement key by design.
- `status` transitions must obey the status table above: `active -> draft` and `archived -> draft` are forbidden.

Archive endpoint sets `status = archived`. It does not alter tenant subscriptions.

v1's provider deliberately shipped no routes. v1.1 must update the provider to call `loadRoutesFrom(...)` for the plan-management routes, register `PlanController` in `services()`, and register the `subscriptions_plans_manage` middleware alias. If no permission provider/checker is bound, the HTTP write surface denies by default.

## CLI surface

Commands mirror the HTTP surface for operators and deployment scripts:

```text
subscriptions:plans:create
subscriptions:plans:update
subscriptions:plans:archive
subscriptions:plans:import-config
subscriptions:plans:list
```

Minimum behavior:

- `create`: accepts `plan_key`, `display_name`, `status`, optional Payvia link, and entitlement JSON from an option or file.
- `update`: updates display fields, status, Payvia link, sort order, and entitlement JSON.
- `archive`: archives by `plan_key`.
- `import-config`: upserts config plans into DB without overwriting existing DB rows unless `--force` is supplied.
- `list`: shows key, display name, status, Payvia link, updated time.

Import rules:

1. Read `config/subscriptions.php` `plans`.
2. For each config plan, create a DB row if absent.
3. Default `display_name` from `plan_key` converted to title case.
4. Copy `payvia_priced_plan` to `payvia_priced_plan_uuid` when present.
5. Set imported rows to `active` unless `--status=draft` is supplied.
6. Do not overwrite existing rows unless `--force`.

## Audit

Plan changes are not tenant lifecycle events, so they do not belong in `subscription_events`.

Use structured application logging via a PSR-3 logger when available. Resolve `Psr\Log\LoggerInterface::class` or `logger` from the container, log at `info` level with message `subscriptions.plan_changed`, and include this payload:

```json
{
  "plan_key": "pro",
  "action": "updated",
  "before": {
    "display_name": "Pro",
    "status": "active",
    "entitlements": {
      "reports.export": true,
      "projects.limit": 50
    }
  },
  "after": {
    "display_name": "Pro",
    "status": "active",
    "entitlements": {
      "reports.export": true,
      "projects.limit": 100
    }
  },
  "diff": {
    "projects.limit": { "from": 50, "to": 100 }
  }
}
```

If no PSR-3 logger is installed, emit one compact structured `error_log()` line. Do not create a new audit subsystem or invent a new activity-log service key in v1.1.

The before/after snapshot for PATCH must be read and written in one transaction so the audit diff reflects the row that was actually updated. v1.1 uses last-write-wins for concurrent admin edits; no optimistic locking or `version` column is introduced until immutable revisions become real.

## Security posture

- The management API fails closed.
- Missing authenticated user, permission manager, permission provider, or allow decision means HTTP plan writes are denied.
- CLI commands are local/operator tools and follow existing CLI security assumptions.
- Entitlements remain commercial gates, not authorization. A plan granting `reports.export` does not bypass aegis/user permissions.

## Operational behavior

### Empty DB table

An empty `subscription_plans` table is safe. Catalog resolution falls back to config plans.

### Wiped DB table

Tenants whose `subscriptions.plan_key` matches config still resolve from config fallback. Tenants on DB-only plan keys resolve empty maps until the DB row is restored. This is acceptable and visible; it should be covered in docs and `subscriptions:plans:list`.

### Archived plans

Archived plans resolve for existing tenants. They cannot be assigned through service/API/CLI flows.

### Draft plans

Draft plans can be edited and previewed in admin UI, but they do not resolve and cannot be assigned.

Edits to active plans take effect immediately. v1.1 has one mutable row per `plan_key`, not a draft-revision workflow for already-live plans. Draft status is only a pre-publish workflow for brand-new plans.

## Decisions

| ID | Decision | Resolution |
|---|---|---|
| MPC1 | Catalog source | DB-managed plans now, config remains seed/fallback. |
| MPC2 | Archived semantics | Archived plans resolve for existing tenants but cannot be assigned. |
| MPC3 | Draft semantics | Draft plans do not resolve and cannot be assigned; draft is pre-publish only, so active/archived cannot transition back to draft. |
| MPC4 | Entitlement value space | Only `bool`, `int >= 0`, and explicit `null`. |
| MPC5 | Plan key mutability | `plan_key` is immutable in v1.1. |
| MPC6 | Payvia link | Nullable FK-less `payvia_priced_plan_uuid`; Payvia remains optional. |
| MPC7 | Cache invalidation | Catalog version includes config hash plus DB max updated timestamp. |
| MPC8 | Audit home | Activity/audit log, not `subscription_events`. |
| MPC9 | Delete behavior | No delete in v1.1; archive only. |
| MPC10 | Permission | HTTP write surface requires `subscriptions.plans.manage` through a subscriptions-owned `PermissionManager::can()` guard and fails closed. |
| MPC11 | Empty entitlements | `{}` is valid and means deny every entitlement key. |
| MPC12 | Assignability | `PlanCatalog::isAssignable()` is the single assignment rule for service, CLI, and HTTP flows. |

## Acceptance checklist

- Config-only deployments continue working with an empty `subscription_plans` table.
- Config-only deployments continue working when the extension is enabled before migration 004 has run.
- DB active plans override config plans with the same `plan_key`.
- DB archived plans override config and keep resolving for existing tenants.
- DB draft plans do not resolve.
- API and CLI reject `active -> draft` and `archived -> draft` status transitions.
- Undefined entitlement keys deny; explicit `null` grants unlimited.
- Empty entitlement maps are accepted and documented as deny-everything plans.
- API and CLI reject invalid entitlement values.
- `PlanCatalog::isAssignable()` is used by start/change-plan and operator assignment paths.
- `import-config` is rejected as a `plan_key`, and `{key}` routes constrain dots with `[a-z0-9._-]+`.
- Archive does not mutate tenant subscriptions.
- PATCH audit reads before state and writes after state in one transaction; concurrent edits are last-write-wins in v1.1.
- `import-config` seeds plans without overwriting unless forced.
- Plan edits change the catalog version fingerprint.
- `subscription_plans.updated_at` is indexed for the version fingerprint query.
- The extension provider loads the v1.1 plan-management routes, registers `PlanController`, and applies the fail-closed `subscriptions_plans_manage` guard.
- Existing checker, middleware, rate-tier bridge, Payvia listener, and overrides require no direct DB-plan awareness.
